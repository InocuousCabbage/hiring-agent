## Summary

Single-commit feature adding editable DOCX as a second attachment alongside the existing PDF for each tailored resume and cover letter. Renderers now return `(pdf_path, docx_path)`, `main.py` unpacks and appends both to the attachments list (4 per job), and `GmailClient` dispatches MIME by suffix with an `octet-stream` fallback. The change is small, focused, and faithful to the spec.

## Findings

1. **[MEDIUM] Duplicate-attachment when PDF converter is unavailable** — `src/pdf_gen/renderer.py:116,147` + `src/main.py:334-339` — When LibreOffice/docx2pdf are both unavailable, the renderer returns `(docx_path, docx_path)` (same `Path` instance). `main.py` then appends that path twice as if it were two different artifacts. Email recipients see two attachments with identical filenames; the body text claims a PDF is attached when it isn't. **Recommended action:** in `main.py`, before extending the attachments list, dedupe via `if resume_pdf != resume_docx:` (only append both when they differ), and/or branch the attachment-note text accordingly. Add a regression test for the degraded-path attachment list (4 if PDF available, 2 if not).

2. **[LOW] Pipeline test reimplements main.main()'s attachment logic instead of importing it** — `tests/test_full_pipeline.py:69-79` — The `_build_attachments` helper is a copy of the loop in `main.main()`. If someone changes main.py (e.g., reorders, dedupes per finding #1, or routes through `compose_digest`) the test stays green while the real path breaks. **Recommended action:** extract `_build_attachments(processed)` into `src/main.py` (or `gmail/digest.py`) and import it from the test. Cheap and obvious.

3. **[LOW] Body-text injection lives at the call site, not in `compose_digest`** — `src/main.py:329-333` — Writer flagged this in handoff risk #3. The note is prepended to `compose_digest(...)`'s output rather than added inside it. Works today, but means any second caller of `compose_digest` (e.g., a future preview-in-CLI path) misses the note. **Recommended action:** move the prefix into `gmail/digest.py::compose_digest`, optionally behind a `has_docx: bool` flag so the note only appears when DOCX is actually attached (also helps finding #1).

4. **[NIT] `r_type` / `cl_type` test-mode print is now misleading** — `src/main.py:262-263,265,267` — The label says "Resume (PDF)" or "Resume (DOCX)" based on the resume_pdf path's suffix — but in degraded mode that label could correctly say DOCX while the next line still always says "Resume (DOCX)". Reader confusion only — not a bug.

5. **[NIT] Bare tuple return type** — `src/pdf_gen/renderer.py:84,124` — `tuple[Path, Path]` is fine but error-prone (order swap at call site is silent). A `NamedTuple` (`class RenderResult(NamedTuple): pdf: Path; docx: Path`) or dataclass would be more self-documenting. Low priority — only two call sites today.

6. **[INFO] Resume renderer test skip is legitimate** — `tests/test_renderer.py:228-258` — `templates/resumes/` contains only `README.md`; `base_resume.docx` is user-supplied + gitignored (confirmed). The `pytest.mark.skipif` guard is correct.

7. **[INFO] `deploy/` audit clean** — Only `deploy/Dockerfile` mentions PDF, in an install-comment for LibreOffice. No hardcoded `.pdf` paths anywhere in `deploy/`.

## Score

**4 / 5** — Approve with minor follow-up.

The implementation is correct, focused, well-tested for the happy path, and stays inside the spec. The two real concerns (duplicate-DOCX in degraded mode + the pipeline test mirroring rather than importing main's logic) are easy fixes and don't break the happy path that Ben actually runs day-to-day on macOS with docx2pdf available. They should be addressed before merge but don't require redesign.

## Recommendation

**APPROVE-WITH-FIXES** — Merge after:
- (Required) Dedup attachments in `main.py` when `resume_pdf is resume_docx` (finding #1), or surface the degraded-mode condition in the body text. Add a test for the degraded-mode attachment count.
- (Strongly recommended) Move `_build_attachments` to a shared location so the test imports it (finding #2).
- (Optional) Move the body-text note into `compose_digest` (finding #3).

Findings 4-7 are advisory only; safe to defer.

This PR is ready for Ben to push as a draft for review/discussion of finding #1 — not ready to merge to main without the degraded-mode dedup or an explicit decision to accept that behavior.
